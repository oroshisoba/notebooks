
# 簡易FP法（簡易Function Point法）

簡易FP法は、**システムの機能量をもとに開発規模を見積もる方法**です。
ソースコード行数ではなく、利用者から見て「どんな機能を提供するか」に着目して見積もるのが特徴です。

## **1. FP法とは**

FP法は、システムの規模を
画面、帳票、ファイル、外部連携などの **機能の数と複雑さ**
から数値化する見積もり手法です。
たとえば、次のような機能を数えます。

| **分類**       | **例**                                        |
| -------------- | --------------------------------------------- |
| 入力機能       | 登録画面、更新画面、検索条件入力              |
| 出力機能       | 帳票、一覧画面、CSV出力                       |
| 照会機能       | 検索結果表示、詳細表示                        |
| 内部データ     | 顧客マスタ、契約データ、取引データ            |
| 外部連携データ | 他システムから参照するマスタ、外部APIのデータ |

これらを「機能」として数え、それぞれに点数をつけて合計します。
この合計値が **FP** です。

---

## **2. 簡易FP法とは**

通常のFP法は、機能の複雑度を細かく判定するため、ある程度の時間と経験が必要です。

そこで、要件定義前や提案段階などで使いやすいように、判定を簡略化したものが **簡易FP法** です。

ざっくり言うと、

> 詳細な複雑度判定を省略し、機能の種類ごとにあらかじめ決めた平均点を使って、短時間で概算規模を出す方法

です。

---

## **3. 何を数えるのか**

簡易FP法でも、基本的には以下の5種類の機能を数えます。

| **区分** | **名称**                     | **内容**                                       | **例**                         |
| -------- | ---------------------------- | ---------------------------------------------- | ------------------------------ |
| EI       | 外部入力                     | システムにデータを入力・更新する機能           | 登録、変更、削除、アップロード |
| EO       | 外部出力                     | 加工・集計して外部に出す機能                   | 帳票、集計表、CSV出力          |
| EQ       | 外部照会                     | 入力条件に応じてデータを参照する機能           | 検索、詳細照会                 |
| ILF      | 内部論理ファイル             | システム内部で管理するデータ群                 | 顧客、契約、注文               |
| EIF      | 外部インターフェースファイル | 他システムが管理し、自システムが参照するデータ | 外部マスタ、外部APIデータ      |

ポイントは、**プログラム数ではなく、利用者から見える機能単位で数える**ことです。

---

## **4. 簡易FP法の計算イメージ**

簡易FP法では、機能ごとに平均的な重みを使います。

例として、以下のような重みを使うことがあります。

| **区分**                          | **平均的な重みの例** |
| --------------------------------- | -------------------- |
| EI：外部入力                      | 4点                  |
| EO：外部出力                      | 5点                  |
| EQ：外部照会                      | 4点                  |
| ILF：内部論理ファイル             | 10点                 |
| EIF：外部インターフェースファイル | 7点                  |

たとえば、あるシステムに以下の機能があるとします。

| **区分** | **件数** | **重み** | **小計** |
| -------- | -------- | -------- | -------- |
| EI       | 10       | 4        | 40       |
| EO       | 5        | 5        | 25       |
| EQ       | 8        | 4        | 32       |
| ILF      | 6        | 10       | 60       |
| EIF      | 3        | 7        | 21       |
| 合計     |          |          | 178 FP   |

この場合、システム規模は **178 FP** と見積もります。

その後、過去実績などを使って、
1 FPあたり何人時・何人日か
を掛け合わせて、工数に変換します。
例：

```text
178 FP × 1.2 人日/FP = 213.6 人日
```

---

## **5. 見積もりの流れ**

簡易FP法の流れは、だいたい以下です。

```text
1. 画面・帳票・外部IF・データ一覧を洗い出す
2. 各機能を EI / EO / EQ / ILF / EIF に分類する
3. 件数を数える
4. 区分ごとの平均重みを掛ける
5. FPを合計する
6. 生産性係数を掛けて工数に変換する
7. 難易度・非機能・テスト・管理工数などを補正する
```

---

## **6. 簡易FP法のメリット**

大きなメリットは、**要件が固まり切っていない段階でも使いやすい**ことです。

| **メリット**                 | **内容**                                       |
| ---------------------------- | ---------------------------------------------- |
| 早く見積もれる               | 画面一覧や機能一覧があれば概算できる           |
| 利用者視点で説明しやすい     | 「画面数」「帳票数」「データ数」と対応しやすい |
| 言語や開発方式に依存しにくい | Java、Python、ローコードなどに左右されにくい   |
| 過去案件と比較しやすい       | FPあたり工数で生産性比較ができる               |
| 提案段階に向いている         | 超概算・概算見積もりに使いやすい               |

---

## **7. 注意点・弱点**

一方で、簡易FP法だけで精緻な見積もりを出すのは危険です。

| **注意点**                         | **内容**                                                 |
| ---------------------------------- | -------------------------------------------------------- |
| 非機能要件を表しにくい             | 性能、可用性、セキュリティ、監査対応などは別途補正が必要 |
| UIの複雑さを拾いにくい             | 単純な登録画面と複雑な業務画面が同じEIになりやすい       |
| バッチ処理やデータ移行に弱い       | 大量データ処理、移行、初期データ整備は別見積もりが必要   |
| 外部連携の難易度を過小評価しやすい | API仕様調整、認証、エラー処理などは別途考慮が必要        |
| 件数の数え方にブレが出る           | 何を1機能とするかのルール統一が必要                      |
| 生産性係数が重要                   | FPを工数に変換する係数が実績に合っていないとズレる       |

特に、最近のWebシステムやクラウドシステムでは、以下はFPとは別枠で見積もることが多いです。

```text
・インフラ構築
・認証・認可
・監視・ログ
・CI/CD
・セキュリティ対応
・性能試験
・データ移行
・外部API調整
・運用設計
・プロジェクト管理
```

---

## **8. どういう場面で使うべきか**

簡易FP法は、特に以下の場面に向いています。

| **場面**               | **向いている理由**               |
| ---------------------- | -------------------------------- |
| 提案前の概算見積もり   | 詳細設計前でも使える             |
| 要件定義前の規模感把握 | 機能一覧から概算できる           |
| 複数案の比較           | A案・B案の機能量を比較できる     |
| 過去案件との比較       | FPあたり工数で妥当性を確認できる |
| 顧客説明               | 機能数ベースなので説明しやすい   |

逆に、詳細見積もりでは、簡易FPだけでなく、画面単位・API単位・バッチ単位・テスト単位の積み上げ見積もりと併用した方が安全です。

---

## **9. 実務での使い方のイメージ**

実務では、以下のように使うのが現実的です。

```text
簡易FP法で概算規模を出す
↓
過去案件の生産性を使って概算工数に変換
↓
非機能・インフラ・データ移行・外部連携を別途加算
↓
リスクバッファを加える
↓
見積もりレンジとして提示する
```

たとえば、

```text
アプリ機能部分：300 FP × 1.5 人日/FP = 450 人日
非機能対応：80 人日
外部連携調整：60 人日
データ移行：50 人日
管理・品質保証：100 人日
リスクバッファ：15%

合計：約850 人日
```

のように、FPだけで完結させず、周辺作業を追加するのが重要です。

---

## **まとめ**

簡易ファンクションポイント法は、

システムの画面・帳票・照会・データ・外部連携などを数えて、機能量から概算規模を出す見積もり方法

です。

提案段階や要件定義前の概算見積もりには便利ですが、非機能要件、外部連携、データ移行、インフラ、セキュリティ、運用設計などは拾いにくいため、**FP見積もり＋別途補正**で使うのが実務的です。
